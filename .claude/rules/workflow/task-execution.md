# 任务执行流程规范

任务实施前必须完成合理性评估与方案确认，未通过时不得直接编码。

## 合理性评估

基于项目现状评估任务内容是否符合最佳实践：

- 任务目标与项目架构、技术栈方向是否一致
- 实现路径是否为最短路径，是否存在更优解
- 是否引入超出任务范围的抽象或为假设的未来需求设计

评估不通过时直接中断，向用户说明不合理的原因并建议更合理的方向。

## 方案设计与确认

合理性评估通过后，输出实施方案，包含：

- **核心逻辑**：实现任务的关键思路和处理流程
- **涉及文件**：需要新增、修改、删除的文件清单
- **修改要点**：每个文件的关键改动说明
- **遗漏补充**：发现用户描述未覆盖但实现必需的内容时，主动指出

方案经用户手动确认后，方可开始实施。

## 编码前置

实施任务前必须充分理解 `.claude/rules/` 目录下的所有规范文档，确保代码符合项目约定。涉及多个规范交叉时以更严格的约定为准。

## 模糊点处理

任务中遇到没有明确逻辑处理的环节，禁止基于假设自行发挥：

- 在对应位置使用 `// TODO` 注释标注此处待人工处理
- 或直接停下来提供方案供选择，不要强行实现

## 任务描述转化

禁止将用户描述的任务内容原封不动复制到代码注释、提交信息或文档中。需要在充分理解后，以符合最佳实践和项目既有风格的方式重新表述。

## 任务完成评审

任务实施完成后，按 `task-completion-review.md` 执行收尾评审。
