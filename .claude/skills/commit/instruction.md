# Git 代码提交执行指令

## 重要提示

当用户表达需要提交代码时，严格按照以下步骤执行 Git 提交流程。所有操作必须遵循项目的 Git 最佳实践。

---

## 执行步骤

### 第一步：查看当前工作区状态

```bash
git status
```

**目的**：了解当前有哪些文件被修改、新增或删除。

---

### 第二步：查看具体的修改内容

```bash
git diff
```

```bash
git diff --staged
```

**目的**：详细分析所有修改内容，为生成准确的提交消息做准备。

---

### 第三步：分析修改内容并生成提交消息

**消息格式要求**：

1. **使用中文**：提交消息必须使用中文表达
2. **简洁明了**：标题行不超过 50 个字符
3. **动词开头**：使用祈使句，说明"做了什么"而非"做了什么"
4. **分类标识**：在消息前添加适当的类型前缀：
   - `feat` - 新功能
   - `fix` - bug 修复
   - `refactor` - 代码重构
   - `docs` - 文档更新
   - `style` - 代码格式调整
   - `test` - 测试相关
   - `chore` - 构建/工具链相关

**标准格式**：

```
<类型>(<范围>): <简短描述>

<详细说明（可选）>
```

**示例**：
```
feat(user): 添加用户注册功能

- 实现了注册接口
- 添加了邮箱验证逻辑
- 确保密码加密存储
```

---

### 第四步：添加所有修改到暂存区

```bash
git add .
```

**注意**：将所有修改内容一次性添加到暂存区，确保所有改动被包含在本次提交中。

---

### 第五步：创建提交

```bash
git commit -m "$(cat <<'EOF'
<根据第三步分析生成的提交消息>

🤖 Generated with [Claude Code](https://claude.com/claude-code)

Co-Authored-By: Claude Sonnet 4.5 <noreply@anthropic.com>
EOF
)"
```

**说明**：
- 使用 HEREDOC 格式确保提交消息格式正确
- 自动包含 Claude Code 的标识信息
- 提交消息必须基于实际的代码修改内容

---

### 第六步：验证提交成功

```bash
git status
```

**预期结果**：工作区应该是干净的（clean working directory），没有任何未提交的修改。

---

### 第七步：推送到远程仓库

```bash
git push
```

**说明**：将本地提交推送到远程仓库，完成整个提交流程。

---

## 执行原则

1. **单次提交**：所有修改必须合并为一次提交，不允许拆分多次提交
2. **工作区干净**：完成后必须确保工作区没有任何未提交的修改
3. **消息准确**：提交消息必须准确反映实际修改的内容
4. **自动推送**：必须在提交成功后自动执行推送操作
5. **完整流程**：严格按照上述七个步骤顺序执行，不得跳过任何步骤

## 错误处理

- 如果提交失败（如 pre-commit hook 错误），必须修复问题后重新尝试提交
- 如果推送失败（如远程有新提交），必须先 pull 再尝试推送
- 如果遇到冲突，必须解决冲突后重新完成提交流程
